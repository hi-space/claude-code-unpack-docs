1. 지시사항 블록 등록

각 지시사항(예: 프로그램 역할, 도구 사용법, Git 규칙)은 독립적인 모듈로 구현되고 중앙 레지스트리에 등록됩니다. 빌드 프로세스가 활성화된 블록들을 우선순위 순서대로 정렬 후 조립하며, 피처 플래그나 세션 구성에 따라 특정 블록만 선택적으로 포함됩니다. 이를 통해 코드를 변경하지 않고도 프롬프트의 행동을 수정할 수 있으며, 불필요한 지시사항을 프롬프트에 포함하지 않아 토큰 사용을 절감합니다.

2. 조건부 포함

각 지시사항 블록은 조건 함수를 포함하여 콘텐츠를 동적으로 결정합니다(예: 플랜 모드 활성화 시만 플랜 블록 포함, 에이전트 비활성화 시 에이전트 블록 스킵). 빈 블록은 자동으로 필터링되어 최종 프롬프트에 포함되지 않습니다. 이 유연성은 다양한 모드와 구성을 지원하면서도 프롬프트 크기를 최소화하고, 캐시 무효화 감지기가 변경사항을 추적하여 불필요한 캐시 재처리를 방지합니다.

3. 성능 경고 이름 규칙

캐시 경계 이후의 접미사 블록은 DANGEROUS_uncachedSystemPromptSection 형태의 변수명으로 표시되어 개발자에게 명확한 경고 신호를 제공합니다. 이 섹션에 내용을 추가하면 매 API 호출마다 재처리되며 비용과 지연 시간 페널티가 발생합니다. 이러한 명시적 네이밍 규칙은 성능 최적화 검토 시 불필요한 접미사 확장을 방지하는 중요한 설계 신호로 작동합니다.

4. 블록 순서의 중요성

캐시 경계 내 블록 순서 변경은 전체 해시를 바꿔서 14-17K 토큰 규모의 캐시 미스를 초래하며, 결과적으로 60% 지연 시간 페널티와 비용 회귀를 발생시킵니다. 이를 방지하기 위해 도구 정의는 항상 알파벳순으로 정렬되며 초기화 순서도 엄격히 관리됩니다. 이러한 엄격한 순서 관리는 높은 캐시 히트율(92-98%)을 유지하고 월 10,000 세션 규모에서 10만 달러 이상의 비용 절감을 가능하게 합니다.

⚡ Optimization

Prompt Caching Strategy

시스템 프롬프트는 캐시 가능한 접두사와 세션별 접미사로 분할되어 정적 부분(약 20K 토큰)을 캐싱함으로써 API 비용을 최대 90% 절감합니다. 캐시 안정성을 위해 12개 벡터(도구 스키마 변경, 피처 플래그, 모델 전환 등)를 모니터링합니다.

1. 안정적인 캐시 구조

캐시된 접두사(~20K 토큰)는 100% 안정성 목표로 설계되어 요청 간에 변하지 않습니다. ID 블록(~100 토큰), 도구 스키마(14-17K), 안전 규칙, 작업 실행 지침, Git 프로토콜을 포함하며, 변경되는 MCP 도구/훅/레포지토리 컨텍스트는 접미사로 분리됩니다. 이러한 분리로 92-98% 캐시 히트율을 달성하고 요청당 18-20K 토큰을 캐싱합니다.

2. 도구 정렬

도구 정의(캐시된 접두사의 가장 큰 섹션, 14-17K 토큰)는 모든 요청에서 동일한 순서를 보장해야 합니다. ToolRegistry.getSchemas()에서 Array.from().sort((a,b) => a.name.localeCompare(b.name))로 알파벳순 정렬을 강제하여 Map 반복 순서 불안정성을 극복합니다. 순서 변경은 해시 불일치로 이어져 캐시 미스와 60% 지연 시간 페널티를 초래하므로 이러한 엄격한 정렬은 필수입니다.

3. 캐시 무효화 감지

CacheBreakDetector(src/prompt/)는 12개 벡터를 각각 모니터링합니다: 도구 스키마 변경, 피처 플래그, 모델 전환, 권한 모드, 프롬프트 버전, 훅 설정 변경 등. 무효화 감지 시 세션의 캐시된 접두사를 삭제하여 다음 요청에서 전체 prefix를 재처리하고 새 캐시 키를 저장합니다. 로깅을 통해 어떤 벡터가 캐시 break를 유발했는지 추적할 수 있습니다.

4. 비용 절감

200 요청 세션 경제 분석: 캐싱 없음 기준 200 × 20K 토큰 × $0.003/1K = $12.00 vs 캐싱 적용 200 × 20K × $0.0003(90% 할인) + 초기 미스 = $1.80 (85% 절감, 세션당 $10.20). 월 10,000 세션 규모에서는 약 $102,000 절감됩니다. 이러한 경제성이 도구 알파벳 정렬, CacheBreakDetector 등 지속적 최적화의 동인이 되며, 캐시 안정성 유지의 중요성을 강화합니다.

⚠ Security

Safety Rules

시스템 프롬프트는 AI 안전(유해한 출력 방지)과 소프트웨어 보안(안전하지 않은 코드 방지)을 모두 다루는 상세한 안전 규칙을 포함합니다. OWASP Top 10을 인식하며, 파괴적 작업은 사용자 확인, Git 특화 규칙은 커밋/푸시 안전을 강제합니다.

Git 특화 안전

1. 보안 취약점 방지

시스템 프롬프트 safetyRulesBlock은 OWASP Top 10을 명시합니다:
- A1 Injection: Bash 도구에 신뢰할 수 없는 변수 검증 필수
- A2 SQL Injection: 매개변수화된 쿼리 사용 지시
- A4 XSS: HTML 생성 새니타이징 필수
모델이 변수 주입 등 위험한 패턴을 작성할 때 시스템 프롬프트가 즉시 감지하고 수정하도록 지시합니다(soft constraint). 권한 모델은 hard constraint 층으로 추가 보호를 제공합니다.

2. 위험한 작업 차단

파괴적 작업(rm -rf, DROP TABLE, git reset --hard, git push --force)은 항상 사용자 확인이 필수입니다. 이중 방어로 구성됩니다:
(1) 시스템 프롬프트에서 모델이 사용자에게 명시적 확인을 요청하도록 지시
(2) 권한 분류기가 HIGH_RISK로 표시하여 UI에서 확인 프롬프트를 강제
이러한 이중 제약으로 우발적인 파괴 작업을 방지합니다.

3. 범위 확인

시스템 프롬프트의 taskExecutionBlock이 모델에게 명확히 지시: 사용자 요청 범위를 엄격히 유지합니다. 예: "오래된 빌드 삭제" 요청 시 old-build 디렉토리만 제거, 전체 시스템 삭제 금지. PermissionChecker가 파일 경로 범위를 검증하고 ToolUseClassifier가 의도를 파악하여 SAFE 표시합니다. 범위를 초과하려는 시도는 Layer 3에서 자동 차단됩니다.

4. Git 설정 보호

git config 명령은 시스템 프롬프트와 권한 모델에서 완전히 차단됩니다. 악의적인 구성 변경은 모든 후속 git 작업(push)을 공격자의 repo로 리다이렉트할 수 있기 때문입니다. gitProtocolBlock에서 모델에게 git config 사용 금지를 명시하고, PermissionChecker가 해당 호출을 감지하면 자동으로 거부합니다. 모든 git 명령 후 상태 검증으로 손상된 구성을 감지할 수 있습니다.

5. Git 훅 보호

--no-verify 플래그 사용은 시스템 프롬프트와 권한 모델에서 무조건 차단됩니다. Pre-commit 훅 우회 시 linting, 비밀 스캔(hardcoded keys), 보안 검증이 모두 무효화되기 때문입니다. gitProtocolBlock이 모델에게 --no-verify 금지를 명시하고, PermissionChecker가 해당 플래그를 감지하면 HIGH_RISK로 표시하여 자동 거부합니다. 이러한 이중 제약으로 배포 파이프라인의 보안 무효화를 방지합니다.

✎ Behavior

Behavioral Directives

시스템 프롬프트에는 Claude Code의 톤, 출력 스타일, 코딩 철학을 지배하는 자세한 지침이 포함되어 있습니다. 이러한 지침은 자의적 스타일 선호도가 아닌 특정 기술적, 경제적 이유로 존재합니다(출력 토큰이 입력의 5배 비용, 캐싱 안정성, 보안 강화).

Output Efficiency

지침 규칙 존재 이유
간결성 요점을 직접 전달. 가장 간단한 접근법 먼저 출력 토큰 비용 = 입력 토큰의 5배. 장황한 응답은 비용이 많이 드는 응답
서론 없음 추론이 아닌 답변/조치로 시작 명령어 에코 및 문맥 설명의 LLM 경향성 방지
채우기 없음 채우기 단어, 서론, 불필요한 전환 건너뛰기 채우기 단어는 의미 있는 값 없이 토큰을 소비
재설명 없음 사용자가 말한 것을 다시 설명하지 마세요 사용자는 이미 자신이 요청한 것을 알고 있음
짧은 문장 한 문장으로 말할 수 있다면 세 문장 사용 금지 터미널 렌더링은 긴 문장을 읽기 어렵게 만듦
이모지 없음 사용자가 명시적으로 요청하지 않는 한 전문적 문맥의 기본값

Coding Philosophy

1. 출력 효율

출력 토큰 비용이 입력의 약 5배이므로 outputEfficiencyBlock이 모델에게 명확히 지시합니다:
- 요점 직접 전달 (서론/문맥 설명 금지)
- 가장 간단한 접근법 먼저
- 불필요한 채우기 단어 제거
- 사용자가 이미 아는 내용 재설명 금지
간결성은 비용 최적화와 사용자 시간 절약(AI 슬롭 방지)을 동시에 달성합니다.

2. 도구 선호도

도구 사용 우선순위가 명시되어 있습니다: Read (cat/head/tail 대체) → Edit (sed/awk 대체) → Write (echo/cat 대체) → Bash (최후 수단). 전용 도구는 오류 처리와 권한 검증의 일관성을 보장합니다. 권한 모델도 Bash보다 Read/Edit/Write를 자동 허용하여 안전성을 확보합니다.

3. Git 안전 규칙

gitProtocolBlock이 모델에게 상세한 지시사항을 제공합니다:
- --force 푸시 금지 (히스토리 손상 위험)
- --amend 금지 (훅 실패 후 이전 커밋 수정 가능성)
- --no-verify 금지 (보안 훅 무효화)
- 비밀(credentials) 커밋 금지 (새 커밋 생성)
- 각 git 작업 후 상태 검증
모든 규칙이 권한 모델 Layer 1 allowlist + Layer 3 분류기로도 강제되어 이중 보안을 제공합니다.

4. 과도한 설계 방지

antiPatternBlock(taskExecutionBlock의 일부)이 명시합니다:
- 불필요한 추상화 금지 (YAGNI — You Ain't Gonna Need It)
- 일회용 작업 헬퍼 생성 금지
- 요청 범위 초과 기능 추가 금지
- 리팩토링 중 인접 코드 수정 금지
- 미사용 코드는 완전 삭제
이 지침들은 실제 production failures 경험에서 파생되었으며, 과도한 엔지니어링을 방지하고 최소 viable change를 선호합니다.

5. 완료 전 검증

taskExecutionBlock의 최종 지침: 완료 주장 전 검증 필수입니다. 빌드/테스트를 실행하고 fresh output을 표시해야 합니다. 가정 불가 — 검증만 가능합니다. 실패를 무시할 수 없으며 근본 원인 진단 후 수정이 필요합니다. 경제성: 초기 검증 비용 << 후속 재작업 비용. 작은 정확한 변경이 큰 clever한 변경보다 우수합니다.
