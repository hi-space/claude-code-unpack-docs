코디네이터 모드

Claude Code의 다중 에이전트 오케스트레이션 시스템. 오케스트레이션 로직이 코드가 아닌 시스템 프롬프트에 존재하여, 코드 변경 없이 전략을 수정할 수 있습니다.

Agent Type Process Model Isolation Communication Use Case
InProcessTeammate Shared V8 Heap AsyncLocalStorage Direct Memory Lightweight parallel tasks
LocalAgent Separate Process Process Boundary File Mailbox IPC Long-running work
RemoteAgent Remote Machine Network Boundary Teleport RPC Distributed compute
LocalShell Subprocess Process Boundary stdin/stdout Shell commands

1. Prompt-Based Orchestration

오케스트레이션 로직이 코드가 아닌 시스템 프롬프트에 존재합니다. 코디네이터 프롬프트는 작업 분해, 워커 할당, 품질 제어, 진행 모니터링, 결과 병합의 5단계를 정의하며, "Do not rubber-stamp weak work" 같은 명시적 품질 관리 지시문을 포함합니다. 코드 변경 없이 시스템 프롬프트만 수정하여 전체 오케스트레이션 전략을 즉시 변경할 수 있습니다.

2. "Do Not Rubber-Stamp" Quality Control

코디네이터는 각 워커의 출력을 비판적으로 검증하며, 불완전하거나 부정확한 작업을 거부하고 수정을 요청합니다. 검증 단계에서는 아키텍처 일관성, 원래 요구사항과의 부합성, 반복 실패 추적을 수행하여 단순한 자동화가 아닌 품질 제어를 코드 레벨에 내장합니다.

3. Task Panel Management

TaskList는 모든 활성 작업자를 메모리에 기록하며 CoordinatorAgentStatus.tsx UI 대시보드에 반영됩니다. 각 태스크는 pending → running → completed/failed 상태를 거치며, heartbeat 타임아웃(5분) 초과 시 stalled 태스크로 감지됩니다. 코디네이터는 TaskList만 확인해도 전체 작업 진행 상황을 파악할 수 있습니다.

4. Worker Agent Lifecycle

각 워커는 자신만의 독립 QueryEngine 인스턴스와 격리된 컨텍스트 윈도우를 받습니다. 워커는 완전한 작업 설명(prompt), 허용된 도구 목록, 부모 AbortController를 수신한 후 독립적으로 작동하며, 최종 결과만 코디네이터에 반환합니다. 워커 간에는 conversation history, tool state, permissions이 공유되지 않습니다.

5. Feature Gate - Compile Time Flag

코디네이터 모드는 COORDINATOR_MODE 컴파일 타임 플래그 뒤에 게이트되어 있으며, 공개 빌드에서는 비활성화됩니다. 플래그가 꺼져있으면 Bun 번들러의 tree-shaking 최적화에 의해 src/coordinator/ 전체 디렉토리가 바이너리에서 제거되어 번들 크기 증가가 없습니다.

워크트리 격리 원리

코디네이터 모드에서 각 에이전트는 .claude-worktrees/{taskId} 경로에 전용 git worktree를 생성하여 독립적으로 작업합니다. 모든 에이전트가 동일한 리포지토리를 공유하지만 각자의 브랜치(team/{taskId})에서 변경하므로, 여러 에이전트가 동일 파일을 동시에 수정해도 git merge conflict가 발생하지 않습니다.

자동 정리 및 병합

에이전트가 작업을 완료하면 변경사항을 자신의 브랜치에 커밋하고, worktree는 자동으로 삭제됩니다. 변경사항이 없는 worktree도 자동 정리되어 디스크 낭비가 없습니다. 최종 병합은 코디네이터가 각 에이전트의 PR을 메인 브랜치로 통합하는 방식으로 처리합니다.

★ SubAgents

서브에이전트 유형

5가지 이상의 특화된 서브에이전트 유형, 각각 특정 목적과 제한된 도구 접근 권한. AsyncLocalStorage 격리와 계층적 AbortController를 통한 안전한 병렬 실행.

Type Available Tools Key Constraints Typical Duration
general-purpose All tools Full capability 30s - 5m
Explore Read-only tools Max 3 parallel, 15m timeout 5-30s per agent
Plan Read-only tools Detailed planning focus 10-60s
claude-code-guide Glob, Grep, Read, WebFetch Info retrieval only 5-15s
statusline-setup Read, Edit Minimal tool access 5-10s
verification All except Agent, Edit, Write, NotebookEdit Read-only, produces verdicts 30s-2m

Tool Access Model: Allowlist, Not Denylist

각 서브에이전트 유형은 allowlist 모델을 사용합니다. 도구는 명시적으로 허용되거나 차단되며, allowlist에 없는 도구는 사용 불가능합니다. 이는 denylist 방식(기본 허용, 특정 도구만 차단)과 대조됩니다. Allowlist는 더 안전하며 새 도구 추가 시 각 에이전트 타입에 대해 명시적 검토가 필요합니다.

1. AsyncLocalStorage Isolation

InProcessTeammate는 Node.js AsyncLocalStorage를 통해 동일 V8 힙 내에서 격리된 컨텍스트를 유지합니다. 별도 프로세스 생성 오버헤드(~40MB 메모리, ms 단위 시작 시간) 없이 안전한 병렬 실행이 가능하며, 각 teammate는 자신만의 conversation history, tool state, permission scope를 가집니다. 100+개 에이전트 동시 실행 가능합니다.

2. File-Based Mailbox IPC

LocalAgent 간 통신에 JSON 파일 기반 Mailbox 패턴을 사용합니다. dotfile lockfile으로 원자적 쓰기를 제어하며(10ms 백오프 재시도), 에이전트 크래시 시에도 mailbox 파일이 손상되지 않아 높은 복원력을 제공합니다. 코디네이터는 timeout 감지 시 새 에이전트를 생성하여 미완료 작업을 이어받을 수 있습니다.

3. Task List as Coordination Protocol

Task List는 단순한 할일 목록이 아닌 분산 조율 프로토콜입니다. 에이전트가 크래시되어도 미완료 태스크는 Task List에 남아있어 다른 에이전트가 작업을 claim하여 이어받을 수 있습니다. 코디네이터는 Task List 상태만 확인해도 pending/in_progress/completed/failed 브레이크다운으로 전체 진행 상황을 파악하며, 에이전트와 코디네이터 간의 직접 의존성이 없어 결합도를 최소화합니다.

4. AbortController Hierarchy

계층적 AbortController 구조를 통해 에이전트 수명 주기를 관리합니다. 부모 에이전트가 취소되면 root AbortController의 abort() 호출이 모든 자식 에이전트의 signal로 전파되며, 각 에이전트는 pending 도구 호출을 취소하고 worktree를 정리한 후 종료됩니다. 깊은 중첩 구조(agent → subagent → sub-subagent)에서도 고아 프로세스가 발생하지 않습니다.

5. Git Worktree Isolation

각 에이전트는 전용 git worktree(예: .claude-worktrees/{taskId})에서 독립적으로 작업하여 여러 에이전트가 동시에 동일 파일을 수정해도 git merge conflict가 발생하지 않습니다. 각 에이전트는 자신의 브랜치(team/{taskId})에서 변경사항을 커밋하고, 에이전트 완료 시 워크트리는 자동으로 정리되며 병합은 코디네이터가 처리합니다.

6. Verification Agent

Verification Agent는 구현 후 테스트 및 검증을 위해 특화되었습니다. 읽기 전용 실행, 적대적 포커스(구현 깨뜨리기 시도), 증거 기반 평가(PASS/FAIL/PARTIAL)를 수행합니다. tengu_hive_evidence GrowthBook 플래그로 제어됩니다.

7. Explore Agent Thoroughness Levels

Explore 에이전트는 세 가지 철저함 레벨을 지원합니다:
quick (특정 디렉토리, 1-2 파일, 5-10초), medium (여러 위치, 5-20개 파일, 15-30초), very thorough (전체 코드베이스, 50+ 파일, 30-60초). 최대 3개의 Explore 에이전트가 병렬로 실행 가능하며, 각 에이전트는 15분 타임아웃을 가집니다.

⚙ KAIROS

KAIROS — 자율 데몬 모드

완전히 구현되었지만 컴파일 타임 플래그로 비활성화된 자율 에이전트 데몬. 현재의 반응형 모델에서 지속적으로 작동하는 백그라운드 에이전트로의 패러다임 전환을 나타냅니다.

⚠️
Unreleased Feature KAIROS는 완전히 구현되었지만 컴파일 타임 플래그 뒤에 게이트되어 있으며, 공개 빌드에서는 사용할 수 없습니다. 최종 테스트와 정책 결정을 기다리고 있습니다.

1. Tick Loop System

사용자 입력에 응답하는 일반 Claude Code와 달리, KAIROS는 주기적 tick 루프로 동작합니다. <tick> 프롬프트가 설정된 간격(예: 30-60초)으로 모델에 전송되며, 각 tick에서 모델은 workspace state, pending events, recent actions을 받아 자율적으로 액션 필요 여부를 판단합니다. 액션 결정 후 결과는 dailyLog에 append되고 다음 tick의 컨텍스트가 됩니다.

2. autoDream Memory Consolidation

autoDream은 유휴 시간(DREAM_THRESHOLD 초 이상)에 자동으로 실행되며, 생물학적 수면 중 기억 통합과 유사합니다. 4단계를 거칩니다: (1) dailyLog에서 최근 관측 로드, (2) 관련 관측 병합, (3) 모순 제거 후 실제 코드로 검증, (4) 모호한 관측을 구체적 사실로 결정화(crystallize). 최종 사실은 MEMORY.md에 append되어 데몬의 지식이 시간이 지날수록 정제됩니다.

3. Event Watcher - GitHub Integration

KAIROS는 GitHub 웹훅을 구독하여 pull_request, pull_request_review, issue_comment, check_suite, push, issues 이벤트를 모니터링합니다. 이벤트는 priority로 분류됩니다: high(CI 실패, 리뷰 changes requested) → 즉시 tick 트리거, medium(새 코멘트) → 다음 tick에 포함, low(정보성) → dailyLog에만 기록. 데몬은 GitHub 이벤트에 자율적으로 반응하여 대기 중 능동적으로 대응합니다.

4. Daily Log - Append-Only Journal

KAIROS는 append-only 일일 로그(dailyLog)를 유지하여 각 tick에서: (1) 수행한 액션과 결과, (2) 코드베이스 관측, (3) 취한 결정과 근거, (4) 수신한 GitHub 이벤트와 처리 방식을 기록합니다. 로그 항목은 type(action/observation/event/dream/error), tickNumber, timestamp, context를 포함하며, 이는 autoDream의 입력이 되고 사용자에게 완전한 감사 추적(audit trail)을 제공합니다.

5. Deployment Status - Compile-Time Gate

KAIROS는 완전히 구현되었으며 150+ 코드베이스 참조를 가진 프로덕션-레디 코드이지만, KAIROS 컴파일 타임 플래그 뒤에 게이트되어 있습니다. 공개 빌드에서는 dead-code-eliminated되며, 최종 테스트와 정책 결정을 대기 중입니다. 내부 Anthropic 빌드에서는 KAIROS_ENABLED=true 환경 변수와 GitHub 토큰으로 활성화됩니다.

🔍
Paradigm Shift KAIROS는 AI 코딩 어시스턴트의 다음 진화에 대한 Anthropic의 비전입니다: 요청 시에만 응답하는 게 아니라 지속적으로 이슈를 감시하고, CI 실패와 의존성 업데이트에 자동 대응하며, 시간이 지날수록 autoDream으로 코드베이스 이해도를 향상시킵니다.
