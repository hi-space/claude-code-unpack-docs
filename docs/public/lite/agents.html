코디네이터 모드

여러 에이전트를 조율해서 큰 작업을 나누고 처리하는 방식입니다. 작업 전략을 코드 수정 없이 지시문만 바꿔서 조정할 수 있습니다.

1. Prompt-Based Orchestration

가장 주목할 만한 설계는 에이전트를 지휘하는 오케스트레이션 로직이 코드가 아니라 코디네이터 프롬프트에 담겨 있다는 점입니다. 프롬프트는 작업 분해, 워커 할당, 품질 검증, 진행 모니터링, 결과 병합의 5단계를 정의하며, "작업의 부정확함을 절대 간과하지 말 것"과 같은 명시적 품질 관리 지시를 포함합니다. 덕분에 코드를 건드리지 않고 프롬프트만 수정해서 전체 전략을 즉시 변경할 수 있습니다.

2. "Do Not Rubber-Stamp" Quality Control

코디네이터는 각 워커의 결과물을 비판적으로 검증하며, 불완전하거나 부정확한 작업을 거부하고 수정을 요청합니다. 검증 단계에서는 아키텍처 일관성, 원래 요구사항과의 부합성, 반복 실패 패턴을 추적하여 단순한 자동화가 아닌 품질 제어를 구조화합니다. 이는 코드 레벨에 내장된 엄격한 검수 원칙입니다.

3. Task Panel Management

TaskList는 모든 활성 워커와 작업 상태를 메모리에 기록하며 UI 대시보드에 실시간으로 반영됩니다. 각 작업은 pending → running → completed/failed 상태를 거치며, 5분 이상 응답이 없는 작업은 자동으로 stalled 상태로 감지됩니다. 코디네이터는 TaskList만 확인해도 전체 작업의 진행 상황, 완료 수, 실패 현황을 한눈에 파악할 수 있습니다.

4. Worker Agent Lifecycle

각 워커는 자신만의 독립 QueryEngine 인스턴스와 격리된 컨텍스트 윈도우를 받습니다. 워커는 작업 설명, 허용된 도구 목록, 부모 AbortController를 수신한 후 독립적으로 작동하며, 최종 결과만 코디네이터에 반환합니다. 워커 간에는 대화 기록, 도구 상태, 권한이 공유되지 않으므로 완전히 격리된 환경에서 작업합니다.

5. Feature Gate - Compile Time Flag

코디네이터 모드는 COORDINATOR_MODE 컴파일 타임 플래그 뒤에 게이트되어 있으며, 공개 빌드에서는 비활성화됩니다. 플래그가 꺼져있으면 Bun 번들러의 tree-shaking 최적화에 의해 코디네이터 디렉토리 전체가 바이너리에서 제거되어 번들 크기가 증가하지 않습니다. 필요할 때만 선택적으로 활성화할 수 있는 유연한 구조입니다.

워크트리 격리 원리

코디네이터 모드에서 각 에이전트는 .claude-worktrees/{taskId} 경로에 전용 git worktree를 생성하여 독립적으로 작업합니다. 모든 에이전트가 동일한 리포지토리를 공유하지만 각자의 브랜치(team/{taskId})에서 변경하므로, 여러 에이전트가 동일 파일을 동시에 수정해도 git merge conflict가 발생하지 않습니다. 이것이 대규모 병렬 작업을 가능하게 하는 핵심 메커니즘입니다.

자동 정리 및 병합

에이전트가 작업을 완료하면 변경사항을 자신의 브랜치에 커밋하고, worktree는 자동으로 삭제됩니다. 변경사항이 없는 worktree도 자동 정리되어 디스크 낭비가 없습니다. 최종 병합은 코디네이터가 각 에이전트의 PR을 메인 브랜치로 통합하는 방식으로 처리하므로, 장기 실행 워크플로우에서도 안정적으로 리소스를 관리할 수 있습니다.

Tool Access Model: Allowlist, Not Denylist

각 서브에이전트 유형은 allowlist 모델을 사용합니다. 도구는 명시적으로 허용되거나 차단되며, allowlist에 없는 도구는 사용 불가능합니다. 이는 기본 허용하고 일부만 차단하는 denylist 방식과 대조됩니다. Allowlist는 더 안전하며 새 도구 추가 시 각 에이전트 타입에 대해 명시적 검토가 필요하므로, 실수와 보안 위험을 크게 줄입니다.

1. AsyncLocalStorage Isolation

InProcessTeammate는 Node.js AsyncLocalStorage를 통해 동일 V8 힙 내에서 격리된 컨텍스트를 유지합니다. 별도 프로세스 생성 오버헤드(약 40MB 메모리, 밀리초 단위 시작 시간) 없이 안전한 병렬 실행이 가능하며, 각 teammate는 자신만의 대화 기록, 도구 상태, 권한 범위를 가집니다. 이 설계로 수십~수백 개의 에이전트를 동시에 실행할 수 있습니다.

2. File-Based Mailbox IPC

LocalAgent 간 통신에 JSON 파일 기반 Mailbox 패턴을 사용합니다. 닷파일 lockfile으로 원자적 쓰기를 제어하며(10ms 백오프 재시도), 에이전트 크래시 시에도 mailbox 파일이 손상되지 않아 높은 복원력을 제공합니다. 코디네이터는 timeout 감지 시 새 에이전트를 생성하여 미완료 작업을 이어받을 수 있습니다.

3. Task List as Coordination Protocol

Task List는 단순한 할일 목록이 아닌 분산 조율 프로토콜입니다. 에이전트가 크래시되어도 미완료 태스크는 Task List에 남아있어 다른 에이전트가 작업을 claim하여 이어받을 수 있습니다. 코디네이터는 Task List 상태만 확인해도 pending/in_progress/completed/failed 브레이크다운으로 전체 진행 상황을 파악하며, 에이전트와 코디네이터 간의 직접 의존성이 없어 결합도를 최소화합니다.

4. AbortController Hierarchy

계층적 AbortController 구조를 통해 에이전트 수명 주기를 관리합니다. 부모 에이전트가 취소되면 root AbortController의 abort() 호출이 모든 자식 에이전트의 signal로 전파되며, 각 에이전트는 pending 도구 호출을 취소하고 worktree를 정리한 후 종료됩니다. 깊은 중첩 구조에서도 고아 프로세스가 발생하지 않습니다.

5. Git Worktree Isolation

각 에이전트는 전용 git worktree(예: .claude-worktrees/{taskId})에서 독립적으로 작업하여 여러 에이전트가 동시에 동일 파일을 수정해도 git merge conflict가 발생하지 않습니다. 각 에이전트는 자신의 브랜치(team/{taskId})에서 변경사항을 커밋하고, 에이전트 완료 시 워크트리는 자동으로 정리되며 병합은 코디네이터가 처리합니다.

6. Verification Agent

Verification Agent는 구현 후 테스트 및 검증을 위해 특화되었습니다. 읽기 전용 실행, 적대적 포커스(구현 깨뜨리기 시도), 증거 기반 평가(PASS/FAIL/PARTIAL)를 수행하며, tengu_hive_evidence GrowthBook 플래그로 제어됩니다. 품질 검증의 마지막 관문입니다.

7. Explore Agent Thoroughness Levels

Explore 에이전트는 세 가지 철저함 레벨을 지원합니다:
- quick: 특정 디렉토리, 1-2 파일, 5-10초
- medium: 여러 위치, 5-20개 파일, 15-30초
- very thorough: 전체 코드베이스, 50+ 파일, 30-60초
최대 3개의 Explore 에이전트가 병렬로 실행 가능하며, 각 에이전트는 15분 타임아웃을 가집니다. 필요에 따라 탐색 깊이를 선택할 수 있습니다.

1. Tick Loop System

사용자 입력에 응답하는 일반 Claude Code와 달리, KAIROS는 주기적 tick 루프로 동작합니다. <tick> 프롬프트가 설정된 간격(예: 30-60초)으로 모델에 전송되며, 각 tick에서 모델은 workspace state, pending events, recent actions를 받아 자율적으로 액션 필요 여부를 판단합니다. 액션 결정 후 결과는 dailyLog에 append되고 다음 tick의 컨텍스트가 됩니다.

2. autoDream Memory Consolidation

autoDream은 유휴 시간에 자동으로 실행되며, 생물학적 수면 중 기억 통합과 유사합니다. 4단계를 거칩니다:
- dailyLog에서 최근 관측 로드
- 관련 관측 병합
- 모순 제거 후 실제 코드로 검증
- 모호한 관측을 구체적 사실로 결정화
최종 사실은 MEMORY.md에 기록되어 데몬의 지식이 시간이 지날수록 정제됩니다.

3. Event Watcher - GitHub Integration

KAIROS는 GitHub 웹훅을 구독하여 PR, 리뷰, 코멘트, CI, 푸시, 이슈 이벤트를 모니터링합니다. 이벤트는 우선순위로 분류됩니다:
- high: CI 실패, 리뷰 변경 요청 → 즉시 tick 트리거
- medium: 새 코멘트 → 다음 tick에 포함
- low: 정보성 → dailyLog에만 기록
데몬은 GitHub 이벤트에 자율적으로 반응하여 대기 중에도 능동적으로 대응합니다.

4. Daily Log - Append-Only Journal

KAIROS는 append-only 일일 로그를 유지하여 각 tick에서 다음을 기록합니다:
- 수행한 액션과 결과
- 코드베이스 관측
- 취한 결정과 근거
- 수신한 GitHub 이벤트와 처리 방식
이 로그는 autoDream의 입력이 되고 사용자에게 완전한 감사 추적을 제공합니다.

5. Deployment Status - Compile-Time Gate

KAIROS는 완전히 구현되었으며 150+ 코드베이스 참조를 가진 프로덕션-레디 코드이지만, KAIROS 컴파일 타임 플래그 뒤에 게이트되어 있습니다. 공개 빌드에서는 dead-code-eliminated되며, 최종 테스트와 정책 결정을 대기 중입니다. 내부 Anthropic 빌드에서는 KAIROS_ENABLED=true 환경 변수와 GitHub 토큰으로 활성화됩니다.

Paradigm Shift KAIROS는 에이전트의 미래를 보여줍니다. 단순히 요청에 응답하는 대신, 계속 모니터링하고 자동으로 문제를 해결하며, 시간이 지날수록 똑똑해집니다.
