3-Layer Permission Model

Claude Code는 모든 도구 호출을 실행 전에 평가하는 3단계 보안 아키텍처를 구현합니다. 자동 모드에서는 보조 Claude 모델이 분류기로서 위험을 평가합니다.

ℹ️
Bash 도구 권한 흐름 아래 3-계층 모델은 Bash 도구의 권한 흐름을 설명합니다. 다른 도구들(Read, Write, Edit 등)은 서로 다른 권한 모델을 사용합니다 — 각 도구의 설명서를 참고하세요. Bash는 임의의 쉘 명령을 실행할 수 있기 때문에 가장 복잡한 권한 시스템을 가지고 있습니다.

1. 1단계: 고정 허용목록

무조건 안전하게 실행할 수 있는 읽기 전용 도구의 하드코딩된 목록입니다: Read, Glob, Grep, ToolSearch, AskUserQuestion, TodoRead. 이 도구들은 시스템 상태를 변경하지 않으므로 추가 검증 없이 자동으로 허용됩니다.
핵심: 1단계는 고속 패스(fast-path)로 기능하여 매번 분류기 호출 오버헤드를 피합니다.

2. 2단계: 사용자 설정 규칙

~/.claude/settings.json에서 특정 도구 패턴에 대한 항상 허용 규칙을 설정할 수 있습니다.
예: Bash: npm test, Write: src/test/** 패턴으로 반복적인 권한 요청을 회피합니다.
중요: 이 단계에서도 deny 규칙은 절대 우회할 수 없습니다 (deny-first 원칙).

3. 3단계: AI Classifier (2-Stage)

자동 모드에서만 활성화되는 보조 분류기 (Claude Sonnet 4.6)가 각 도구 호출의 위험을 평가합니다.
1단계: 빠른 판단 → 2단계: 불확실한 경우 깊이 있는 분석을 수행합니다.
핵심 설계: 분류기는 사용자 메시지와 도구 호출만 받습니다 (어시스턴트 추론 제외) - 이는 모델의 "설득"으로부터 독립적인 판단을 보장합니다.
가용성: 분류기는 오직 Anthropic 공식 빌드에서만 활성화됩니다. 타사 빌드에서는 분류기가 { matches: false }를 반환하므로 3단계가 실질적으로 작동하지 않습니다.

4. 위험 카테고리 (6가지)

분류기는 각 도구 호출을 6개 차원에서 평가합니다:

  • 범위 확대 - 사용자 요청 초과 (예: 파일 읽기 요청 → 전체 삭제)
  • 비신뢰 인프라 - 원격 저장소, 외부 API, CI/CD 설정 변경
  • 데이터 누출 - 민감 정보 접근 (.env, 크레덴셜, 개인키)
  • 권한 상승 - sudo, 높은 권한 요구, 시스템 리소스 접근
  • 운영 위험 - 시스템 안정성 영향 (rm -rf, 프로세스 종료)
  • 기타 위험 - 일반 안전 우려

5. Context Stripping (의도적 축소)

분류기 API 요청에 포함되는 대화 컨텍스트는 의도적으로 필터링됩니다:
포함: 사용자 메시지만
제외: 어시스턴트 추론 텍스트, 이전 도구 결과
이는 분류기가 모델의 "설득력 있는 추론"으로부터 독립적인 판단을 하도록 하며, 주요 설계 혁신입니다.

6. Dangerous Permission Stripping

자동 모드 진입 시 위험한 권한 규칙(dangerous: true로 표시된)을 자동으로 제거합니다.
이는 사용자가 수동 모드에서 광범위한 권한을 설정해도 자동 모드에서는 AI Classifier 판단만 따르도록 보장합니다.
안전 보장: 자동 모드 종료 시 원래 권한이 자동 복원됩니다.

7. Denial Tracking Fallback

분류기가 반복적으로 같은 유형의 도구 호출을 거부하는 패턴이 감지되면 fallback이 작동합니다.
대신 사용자에게 직접 확인을 요청하여 분류기의 과도한 보수성을 보정합니다.
효과: 분류기가 과하게 엄격한 경우를 사용자가 오버라이드할 수 있는 escape hatch입니다.

샌드박스 격리

Bash 도구가 샌드박스 환경에서 동작하며 다음의 보호 메커니즘을 적용합니다:

보호 메커니즘 설명
파일시스템 격리 워크스페이스 디렉토리로 제한, .env와 credentials 같은 보호된 경로 차단
네트워크 격리 아웃바운드 연결 제한, 신뢰할 수 있는 호스트 화이트리스트
타임아웃 적용 기본 2분, 최대 10분으로 runaway 프로세스 방지
환경 변수 정제 비정상적인 환경 변수 제거

추가 Bash 보안 계층

3-계층 권한 모델 외에도 Bash 명령은 추가 검증 검사를 거칩니다:

Safe Wrapper 제거

안전한 래퍼 프로그램으로 시작하는 명령은 권한 매칭 전에 정규화됩니다:
timeout, time, nice, nohup, stdbuf 등의 래퍼가 제거됩니다.
또한 명령 시작의 환경 변수 할당(예: NODE_ENV=production npm test)도 안전한 변수 이름을 사용하면 제거됩니다.
예: timeout 30 npm test는 권한 매칭을 위해 npm test로 정규화되므로 deny 규칙을 우회할 수 없습니다.

출력 리다이렉션 검증

Bash 명령의 출력 리다이렉션(>, >> 등)은 별도로 검증됩니다:
시스템 파일로의 리다이렉션 차단 — 예: > /etc/passwd
워크스페이스 외 리다이렉션 승인 필요
이는 공격 방지: cat /etc/shadow > /etc/passwd — 비록 cat

복합 명령 보안

단일 Bash 호출에 여러 명령이 포함된 경우(;, &&, ||, 파이프, 줄바꿈으로 구분):
각 서브명령이 개별적으로 검사됩니다
복합 cd + git 차단 — bare repository RCE 공격 방지. 예: cd /malicious/dir && git status — 악의적인 디렉토리가 명령을 실행하도록 설정된 bare git repo를 포함할 수 있습니다.
deny 규칙이 모든 서브명령에 적용되므로 첫 번째 명령만 검사하지 않습니다.

Bash 의미 검사

권한 시스템은 악의적인 명령을 숨길 수 있는 bash 구문 패턴을 검증합니다:
명령 치환 — $(...), 백틱 패턴은 위험한 구성을 포함하면 플래그됩니다
eval 및 source — 강화된 검사 대상
이스케이프된 연산자 정규화 — 예: rm\ -rf는 실제 형태로 정규화되어 매칭됩니다.

🔒
원격 설정 분류기 동작을 GrowthBook 피처 플래그로 원격 수정 가능합니다. Anthropic이 새 버전을 푸시할 필요 없이 모든 Claude Code 설치에서 권한 모델의 민감도를 조정할 수 있습니다.
▻ Infrastructure

Bash Sandbox

Bash 도구 실행 환경의 격리 및 제한 메커니즘

1. 파일시스템 격리

모든 파일 작업이 워크스페이스 디렉토리로 제한되며, 절대 경로나 심볼릭 링크로의 이스케이프 시도는 차단됩니다.
보호 목록: .env, credentials.json, .ssh, .git 등 민감한 경로는 읽기/쓰기 거부
우회 불가: 경로 해석이 권한 검사 전에 일어나므로 동적 우회가 불가능합니다.

2. 네트워크 격리

기본 정책: 모든 아웃바운드 연결 차단 (deny-by-default).
명시적 화이트리스트: npm, pip, 외부 API 호출 필요 시 사용자가 설정에서 허용 호스트를 지정합니다.
효과: 악성 코드가 데이터 유출, C2 통신, 랜섬웨어 다운로드를 시도해도 네트워크 계층에서 차단됩니다.

3. 타임아웃 및 리소스 제한

모든 Bash 명령에 타임아웃 강제: 기본 2분 → 최대 10분까지 사용자가 연장 가능.
리소스 제한: CPU, 메모리, 프로세스 수 상한선으로 runaway 프로세스 방지
효과: 무한 루프, fork bomb, 메모리 폭탄 공격이 Bash 도구 내에서 격리됩니다.

★ Defense

Anti-Distillation Mechanism

경쟁사가 API 트래픽을 녹화/재생하여 Claude Code의 능력을 디스틸레이션하는 것을 방지하기 위한 다중 레이어 시스템

1. 가짜 도구 주입

클라이언트 부분 (신호만): ANTI_DISTILLATION_CC 컴파일 플래그 + GrowthBook 런타임 플래그 모두 활성화 시 API 요청에 anti_distillation: ['fake_tools'] 파라미터를 포함합니다.
서버 부분 (실제 주입): 서버가 응답에 실제 도구 옆에 디코이 도구 정의를 주입합니다 (완전히 동일한 JSON Schema).
핵심: 클라이언트는 신호만 보냅니다. 실제 가짜 도구 생성 및 주입은 서버에서만 발생합니다.
독성화 효과: 트래픽을 캡처한 공격자의 모델은 가짜 도구를 학습하여 실제 환경에서 오작동합니다 (canary trap).

2. Two-Gate Authorization (컴파일 + 런타임)

Gate 1 (컴파일 타임): ANTI_DISTILLATION_CC 플래그 (빌드 시 설정)
Gate 2 (런타임): GrowthBook tengu_anti_distill_fake_tool_injection (원격 제어)
설계 철학: 둘 다 열려야만 작동 - 컴파일 플래그는 제3자 빌드에서 완전히 제거(dead-code elimination), GrowthBook은 새 빌드 없이 긴급 비활성화 가능합니다.

3. First-Party Session Detection

가짜 도구 주입은 Anthropic 공식 CLI 세션 (Client Attestation 검증)에서만 활성화됩니다.
대상: 공식 바이너리에서의 정품 API 호출만
효과: 비인가 클라이언트는 보호를 받지 않으므로 Anthropic 입장에서는 비용 절감, 공격자는 탐지됨.

4. 추론 요약 (서버 사이드)

완전히 서버 사이드: Claude Code 클라이언트는 요약 로직을 포함하지 않습니다.
도구 호출 사이 어시스턴트의 상세 추론 텍스트를 Anthropic 서버에서 자동으로 핵심 요약으로 압축합니다.
프로세스: ConnectorTextBuffer 수집 → SummarizationEngine 압축 → CryptoSigner 서명 첨부
클라이언트의 역할: 요청 파라미터(anti_distillation: ['fake_tools'] 등)를 설정하여 서버에 보호 적용을 신호합니다.
효과: 네트워크 레벨에서 캡처된 데이터는 요약본만 포함 - 상세 추론 체인은 보호되어 학습 가치 저하.

5. Combined Defense Matrix (3-Layer)

Layer 1 (전송): Client Attestation DRM - 비인가 API 접근 차단
Layer 2 (애플리케이션): 가짜 도구 주입 - 캡처된 도구 스키마 독성화
Layer 3 (서버): 추론 요약 - 캡처된 추론 체인 가치 저하
결과: 공격자는 (1) API 접근 불가 또는 (2) 독성 데이터 또는 (3) 요약본만 획득.

🔒 Cryptography

Client Attestation DRM

API 호출을 위한 DRM으로 요청이 위조되거나 수정된 클라이언트가 아닌 정품 Claude Code 바이너리에서 온 것인지 확인

1. Zig 레벨 구현 (JS 런타임 아래)

해시 계산이 Bun의 네이티브 Zig 레이어(HTTP 스택)에서 실행되므로 JavaScript 런타임이 접근 불가합니다.
우회 불가능: JS 디버거, 몽키패칭, fetch() 오버라이드 모두 작동 안 함
수정 비용: 해시 로직 변경 시 Bun 전체 재컴파일 필요 (source 접근 + Zig 컴파일러 필요).

2. 기술 상세 (구현 세부)

플레이스홀더: JS에서 cch=00000 헤더 설정
계산: Zig 레이어에서 HMAC-SHA256 계산, 요청 바디 + 타임스탬프 기반
제어: 컴파일 플래그 NATIVE_CLIENT_ATTESTATION + 환경 변수 CLAUDE_CODE_ATTRIBUTION_HEADER=disabled + GrowthBook tengu_attribution_header (3단계 제어).

3. 왜 Zig인가? (설계 이유)

격리: JS 런타임 아래 실행되므로 디버거/패칭 불가능
성능: 네이티브 HMAC 계산 → 무시할 수준의 레이턴시 추가
변조 저항: 컴파일된 바이너리 내 secret key 임베딩 + Zig 재컴파일 필수
핵심 혁신: 암호화 검사를 런타임 전에 컴파일 바이너리에 고정.

4. 우회 벡터 (의도적 탈출 해치)

개발용: 환경 변수 CLAUDE_CODE_ATTRIBUTION_HEADER=disabled 설정 시 헤더 계산 스킵
긴급 전역 비활성화: GrowthBook tengu_attribution_header=false로 새 빌드 없이 원격 제어
설계 철학: 이 두 escape hatch는 의도적 - 개발/테스트 유연성과 배포 후 버그 대응을 위함.

5. 안티 디스틸레이션과의 관계

클라이언트 인증은 3-레이어 안티 디스틸레이션 체계의 전송 레벨 게이트입니다.
상호 보완:
• 클라이언트 인증 → 비인가 접근 차단 (수비)
• 가짜 도구 → 캡처 도구 스키마 독성화 (독성)
• 추론 요약 → 캡처 추론 체인 가치 저하 (감소)
공격자가 모든 3개 방어를 동시에 우회할 수 없습니다.

🙇 Privacy

Undercover Mode

외부 또는 공개 컨텍스트에서 동작할 때 Claude Code의 출력에서 모든 내부 Anthropic 메타데이터를 제거

1. 제거되는 정보 (6가지)

공개 또는 오픈소스 레포지토리에서 자동 실행 시 다음 우발적 노출 방지:

  • 내부 모델 코드네임 (Capybara, Fennec, Numbat, Tengu)
  • 미출시 모델 버전 번호
  • 내부 Anthropic 레포지토리 이름
  • 내부 인프라 및 도구 참조
  • Slack 채널/이슈/메세지
  • Anthropic 내부 단축 링크

2. 활성화 메커니즘 (3가지)

자동 (기본값): 외부 빌드에서 자동으로 ON
강제: 환경 변수 CLAUDE_CODE_UNDERCOVER=1 설정
구현: 외부 빌드에서 필터링 로직이 dead-code elimination으로 제거되어, 호출 시 항상 trivial return (no-op)만 수행.

3. 비활성화 불가 (강제)

외부 빌드에서는 언더커버 모드 비활성화 불가능 (우회 방법 없음).
구현: 필터링 로직이 dead-code elimination으로 완전 제거 → 호출 불가능한 no-op으로 축소
설계 철학: 컴파일 타임 제거로 "비활성화할 코드 자체 없음" - 가장 강력한 보안.

4. 다른 보안 메커니즘과의 관계

언더커버 모드는 4-계층 보안 스택의 정보 누출 방지 레벨입니다:
Layer 1: 권한 모델 (Permission) - 비인가 액션 차단
Layer 2: Client Attestation - 비인가 API 접근 차단
Layer 3: Anti-Distillation - 학습 데이터 도용 방지
Layer 4: Undercover Mode - 내부 메타데이터 유출 방지 (이 계층).
