3-Layer Permission Model

모든 도구 호출이 고정 허용목록 → 사용자 규칙 → AI 분류기 순서로 3단계를 거쳐 검증됩니다. 안전한 읽기 도구는 고속 패스로 빠르게 통과하고, 위험한 작업은 당신에게 확인을 요청합니다.

ℹ️
Bash 도구 권한 흐름 아래 3-계층 모델은 Bash 도구의 권한 흐름을 설명합니다. 다른 도구들(Read, Write, Edit 등)은 서로 다른 권한 모델을 사용합니다 — 각 도구의 설명서를 참고하세요. Bash는 임의의 쉘 명령을 실행할 수 있기 때문에 가장 복잡한 권한 시스템을 가지고 있습니다.

1. 1단계: 고정 허용목록

Read, Glob, Grep 같은 읽기 전용 도구들은 시스템을 변경할 수 없으므로 항상 자동으로 허용됩니다. 이 허용목록은 하드코딩되어 있어서, 매번 AI 분류기를 호출할 필요 없이 빠르게 실행되는 고속 패스(fast-path)로 작동합니다. 이 설계는 파일 읽기나 검색 같은 안전한 작업의 성능을 크게 개선합니다.

2. 2단계: 사용자 설정 규칙

설정 파일에서 특정 도구 패턴(예: npm test, src/test/**)을 "항상 허용" 규칙으로 지정하면 매번 승인 요청을 받지 않습니다. 예를 들어 테스트를 자주 실행하거나 특정 폴더에만 파일을 쓰는 반복 작업을 설정할 수 있습니다. 하지만 거부 규칙(deny)은 절대 우회될 수 없어서 보안을 유지합니다.

3. 3단계: AI 분류기

자동 모드에서만 활성화되는 별도의 AI(Claude Sonnet)가 각 도구의 위험도를 독립적으로 판단합니다. 이 분류기는 당신의 요청과 도구 호출만 받고, 메인 AI의 추론 내용은 보지 않아서 AI가 자신의 판단을 '설득'할 수 없습니다. 안전한 명령은 바로 실행하고, 의심스러우면 당신에게 확인을 요청합니다.

4. 6가지 위험 카테고리

분류기는 범위 확대(요청 초과), 비신뢰 인프라(원격 저장소), 데이터 누출(민감 정보), 높은 권한(sudo), 시스템 영향(부팅), 기타 우려 등 6가지 차원에서 위험을 평가합니다. 이러한 체계적 분류는 단순한 안전/위험 판단을 넘어서 위험의 성격을 정확히 파악할 수 있게 합니다.

5. 독립적 검증

분류기는 메인 AI의 추론을 보지 않고 오직 사용자의 원래 요청과 실행할 도구만 봅니다. 이렇게 설계한 이유는 메인 AI가 "이 도구는 안전해" 라고 설득력 있게 설명하더라도 분류기는 객관적으로 판단하도록 보장하기 위함입니다. 이는 AI 시스템의 '설득'을 방어하는 핵심 보안 기법입니다.

6. 자동 모드 보호

자동 모드에서는 당신이 수동 모드에서 설정한 광범위한 권한을 자동으로 무시합니다. 위험한 권한으로 표시된 규칙들이 제거되어 오직 AI 분류기의 판단만 적용되므로, 더 엄격한 검증을 받게 됩니다. 자동 모드를 종료하면 원래 권한 설정이 자동으로 복원됩니다.

7. 반복 거부 시 사용자 오버라이드

분류기가 같은 유형의 도구 호출을 반복적으로 거부하는 패턴이 감지되면, 자동으로 당신에게 직접 물어보는 모드로 전환됩니다. 이는 분류기가 과도하게 보수적일 때 당신이 안전성을 판단하고 진행할 수 있는 탈출구(escape hatch)를 제공합니다. 최종 결정권은 항상 당신에게 있습니다.

샌드박스 격리

Bash 도구가 샌드박스 환경에서 동작하며 다음의 보호 메커니즘을 적용합니다:

보호 메커니즘 설명
파일시스템 격리 워크스페이스 디렉토리로 제한, .env와 credentials 같은 보호된 경로 차단
네트워크 격리 아웃바운드 연결 제한, 신뢰할 수 있는 호스트 화이트리스트
타임아웃 적용 기본 2분, 최대 10분으로 runaway 프로세스 방지
환경 변수 정제 비정상적인 환경 변수 제거

추가 Bash 보안

Bash 명령은 권한 검증 외에도 여러 추가 검사를 거칩니다.

래퍼 명령 정규화

timeout, time, nice 같은 래퍼 명령이나 NODE_ENV=production npm test 같은 환경 변수 할당을 자동으로 제거하고 실제 명령(npm test)으로 정규화합니다. 이렇게 하면 공격자가 금지된 명령을 래퍼로 감싸서 거부 규칙을 우회하려는 시도를 방지할 수 있습니다.

출력 리다이렉션 검증

파일 출력 리다이렉션(>, >>)의 대상도 별도로 검증됩니다. 예를 들어 cat /etc/shadow > /etc/passwd는 비록 cat

복합 명령 검증

세미콜론(;), &&, || 같은 연결자로 묶인 여러 명령은 각각 개별적으로 검증됩니다. 예를 들어 안전한 첫 번째 명령(cdcd /malicious/dir && git status)도 감지되어 차단됩니다.

Bash 구문 검증

명령 치환($(...)), eval, source 같은 고급 bash 기능과 백틱, 이스케이프 패턴을 감지합니다. 이런 기법들은 위험한 명령을 숨기는 데 자주 사용되므로, 감지되면 강화된 검사를 받거나 거부됩니다.

원격 제어 Anthropic은 새 버전 업데이트 없이 보안 정책을 조정할 수 있습니다.
▻ Infrastructure

Bash Sandbox

Bash 명령을 파일시스템, 네트워크, 리소스 제한으로 완전히 격리된 환경에서만 실행합니다. 이는 악성 코드의 확산과 시스템 전체 손상을 방지합니다.

1. 파일시스템 격리

모든 파일 작업이 워크스페이스 디렉토리로 제한되며, 절대 경로나 심볼릭 링크를 이용한 이스케이프 시도는 차단됩니다. .env, credentials.json, .ssh, .git 같은 민감한 경로는 추가로 보호되어 읽기/쓰기가 거부됩니다. 경로 해석이 권한 검사보다 먼저 일어나므로 동적 우회 공격이 불가능합니다.

2. 네트워크 격리

기본 정책은 모든 아웃바운드 연결 차단(deny-by-default)입니다. npm 설치나 외부 API 호출이 필요하면 설정 파일에서 허용 호스트를 명시적으로 지정해야 합니다. 이 설계는 악성 코드의 데이터 유출, C2 통신, 랜섬웨어 다운로드 등을 네트워크 계층에서 원천적으로 차단합니다.

3. 타임아웃 및 리소스 제한

모든 Bash 명령은 기본 2분, 최대 10분 타임아웃으로 실행되며, CPU, 메모리, 프로세스 수에도 상한선이 설정됩니다. 이는 무한 루프, fork bomb, 메모리 폭탄 같은 도스 공격이 Bash 도구 내에서 격리되도록 합니다.

★ Defense

Anti-Distillation Mechanism

API 트래픽 도용 또는 클라이언트 조작을 통해 경쟁사가 Claude Code의 기술을 학습하지 못하도록 가짜 도구, 암호화 서명, 추론 요약으로 다층 방어합니다.

1. 가짜 도구 주입 (Canary Trap)

API 응답에 실제 도구 옆에 완전히 동일한 JSON 스키마를 가진 가짜 도구를 서버에서 주입합니다. 트래픽을 가로챈 공격자가 이 데이터로 자신의 모델을 학습시키면, 가짜 도구를 배우게 되어 실제 환경에서 오작동하게 됩니다. 이는 보안 분야의 전형적인 "canary trap" 방어입니다.

2. 이중 게이트 (컴파일 + 런타임)

보호가 작동하려면 빌드 타임의 컴파일 플래그와 런타임의 원격 제어 플래그 두 가지가 모두 활성화되어야 합니다. 비인가 버전은 컴파일 플래그가 제거되므로 자동으로 비활성화됩니다. 긴급 상황에서는 새 빌드 없이 원격 플래그만으로 보호를 비활성화할 수 있습니다.

3. 정품 세션만 보호

가짜 도구 주입은 오직 Anthropic 공식 CLI에서의 정품 API 호출에만 작동합니다. 비인가 클라이언트는 이 보호를 받지 않으므로, Anthropic 관점에서는 비용을 절감하고 공격자는 탐지되는 이점이 있습니다.

4. 추론 내용 요약 (서버 사이드)

클라이언트는 요약 로직을 포함하지 않습니다. 대신 Claude 코드의 상세한 추론 텍스트를 Anthropic 서버에서 자동으로 핵심 요약으로 압축합니다. 이렇게 하면 트래픽을 도용한 공격자가 얻을 수 있는 추론 데이터도 요약본만 포함되어 학습 가치가 크게 저하됩니다.

5. 3단계 방어 연계

순차적으로 작동하는 3단계 방어입니다:
- 1단계: 인증 검증 — 정품 API 접근 필수
- 2단계: 가짜 도구 주입 — 통과해도 독성 데이터 포함
- 3단계: 추론 요약 — 원본 대신 요약본만 제공
이 모든 방어를 동시에 우회할 수 없습니다.

🔒 Cryptography

Client Attestation DRM

Anthropic 공식 Claude Code에서만 API 요청을 인증합니다. 암호화 서명과 네이티브 레이어 검증으로 비인가 클라이언트나 조작된 요청은 거부되며, 학습 데이터 보호 메커니즘이 정품 요청에만 적용됩니다.

1. Zig 네이티브 레이어 (JS 런타임 아래)

해시 계산이 Bun의 네이티브 Zig 레이어(HTTP 스택)에서 실행되므로 JavaScript 런타임이 접근할 수 없습니다. 디버거, 몽키패칭, fetch() 오버라이드 같은 JS 수준의 공격이 모두 무효합니다. 이 설계는 추상화 계층 아래에서 검증을 수행하여 우회를 어렵게 만듭니다.

2. HMAC-SHA256 암호화 서명

각 API 요청에 Zig 레이어에서 계산한 HMAC-SHA256 서명을 헤더로 첨부합니다. 서명은 요청 바디와 타임스탬프를 기반으로 계산되므로 변조를 감지할 수 있습니다. 서버가 서명을 검증하여 정품 클라이언트로부터의 요청만 인식합니다.

3. 바이너리 레벨 고정

암호화 검증 로직은 컴파일된 바이너리에 내장되어 있으며, 암호화 키도 바이너리 자체에 임베딩됩니다. 런타임에 로직을 변경하거나 키를 추출하려면 Zig 코드를 수정하고 전체 바이너리를 재컴파일해야 합니다.

4. 원격 제어 및 긴급 비활성화

검증을 비활성화할 수 있는 환경 변수와 원격 제어 플래그(GrowthBook)가 제공됩니다. 이는 개발 중이거나 긴급 상황에서 Anthropic이 새 빌드 없이 빠르게 대응할 수 있게 합니다.

5. 다층 방어의 첫 번째 계층

클라이언트 인증 (정품 확인) → 가짜 도구 주입 → 추론 요약 순으로 연쇄 작동합니다. 정품 요청만 통과하면 이후 단계의 가짜 도구와 요약으로 추가 보호가 적용됩니다. 이렇게 여러 계층을 조합하면 각각을 우회해야 하는 부담을 늘립니다.

🙇 Privacy

Undercover Mode

비공식 빌드나 공개 환경에서 스크린샷과 로그에 내부 모델명, 미출시 버전, 인프라 정보 등이 자동으로 필터링됩니다. 이는 Claude Code의 기술 스택이 실수로 노출되는 것을 방지합니다.

1. 자동 필터링 레이어

공개 환경(unofficial build)에서는 내부 모델명, 미출시 버전, 내부 레포명, 인프라 참조, 개인 링크, 개발자 정보 등이 자동으로 제거됩니다. 이는 Claude Code의 스크린샷이나 로그가 공개되어도 민감한 내부 정보가 노출되지 않도록 합니다.

2. 비공식 빌드에서 자동 활성화

비공식 빌드(unofficial build)에서는 기본적으로 이 모드가 자동으로 켜집니다. 또한 환경 변수로 공식 빌드에서도 강제로 활성화할 수 있으므로, 테스트나 개발 단계에서 내부 정보 노출을 방지할 수 있습니다.

3. 비활성화 불가 (비공식 빌드)

비공식 빌드에서는 이 필터링을 비활성화할 수 없습니다. 필터 로직이 컴파일 타임에 고정되므로 런타임에 누군가가 우회하려고 시도해도 모든 출력이 자동으로 필터링됩니다.

4. 4단계 보안 체계의 최종 계층

권한 검증(도구 실행 제어) → 클라이언트 인증(정품 확인) → 학습 데이터 보호(가짜 도구, 요약) → 정보 누출 방지(필터링)로 순차 작동합니다. 각 계층이 다른 공격 벡터를 담당하므로, 하나의 방어가 뚫리더라도 다음 계층이 보호합니다.
