3-Layer Memory Architecture

컨텍스트 엔트로피(대화가 길어질수록 품질이 저하되고 환각이 증가하는 현상)를 방지하기 위해 설계된 3단계 시스템입니다. 장시간 코딩 세션에서 AI가 이전 결정을 참조하고 일관성 있게 진행할 수 있도록 구조화됩니다.

핵심 문제: 대화가 길어지면 중요한 정보가 손실되고, AI가 이전 결정을 잊거나 모순된 답변을 합니다.

1단계: MEMORY.md 포인터 시스템

항상 메모리에 있는 가벼운 인덱스. AI가 필요할 때 세부사항을 찾아 읽을 수 있습니다.

속성 설명
형식 Markdown 파일
크기 한 줄에 ~150자
로딩 항상 메모리에 있음
역할 인덱스 / 포인터

2단계: 자동 컨텍스트 압축

대화가 길어지면 오래된 메시지를 자동으로 요약해 공간을 절약합니다.

3단계: 토큰 예산 관리자

메모리 공간을 똑똑하게 나눠씁니다. 시스템, 도구, 대화 등이 각각 필요한 만큼 자동으로 할당됩니다.

1. MEMORY.md 포인터 시스템

컨텍스트 윈도우에 항상 로드되는 경량 포인터 인덱스로, 줄당 약 150자 형식입니다. 저장소 역할이 아니라 인덱스 역할을 하므로, 프로젝트 규모와 무관하게 약 500-1K 토큰만 차지하면서 전체 지식의 지도를 제공합니다. 모델이 필요할 때 Read/Grep으로 세부사항을 검색하는 구조입니다.

2. 자동 컨텍스트 압축

대화 이력이 예산을 초과할 때 오래된 메시지를 자동으로 요약합니다. 중요한 결정과 파일 변경 내역은 보존하면서 장황한 설명과 도구 출력은 제거하여 정보 손실을 최소화합니다. 압축된 메시지에서 새로운 사실을 추출하여 MEMORY.md에 추가하므로, 정보가 단순히 축소되지 않고 더 구조화된 포인터로 업그레이드됩니다.

3. 토큰 예산 관리자

약 1M 토큰 컨텍스트를 동적으로 할당합니다:
- 시스템 프롬프트 약 20-25K
- 도구 스키마 14-17K
- MEMORY.md 약 500-1K
- 대화 이력 약 900-950K
- 응답 예비 약 8-10K
실시간으로 예산 초과를 감지하면 자동으로 대화를 압축하여 장시간 세션도 안정적으로 유지됩니다.

4. Self-Healing Feedback Loop

세 층이 함께 작동하여 컨텍스트 품질이 저하되지 않고 오히려 개선됩니다. MEMORY.md가 항상 로드되므로 모델이 이전 결정을 참조 가능하고, 모순 감지 시 즉시 보정됩니다. 압축 → 새 사실 추출 → MEMORY.md 업데이트의 피드백 루프로, 장시간 세션도 구조화되고 일관성 있게 진행됩니다.

5. 능동적 지식 관리 (미출시)

현재 시스템은 수동적 자가 치유 — 컨텍스트 압박 시에만 트리거됩니다. 미출시 KAIROS 데몬 모드의 자동 기능은 이를 능동적 지식 관리로 진화시킵니다. 유휴 시간에 자동으로 관측 병합(중복 제거), 모순 제거(일관성 유지), 인사이트 결정화(핵심 발견 강화)를 수행하며, 세션 간 학습을 가능하게 합니다.

⚡ Compaction

6-Stage Context Compaction

필요한 최소한의 압축만 적용하여 정보 품질을 최대한 보존하면서 컨텍스트 제한을 유지하는 점진적 압축 시스템입니다. 각 레이어는 필요할 때만 활성화되어, 시간 기반에서 모델 기반, 그리고 긴급 모드까지 단계적으로 발동됩니다.

메모리 할당

구성 요소 Token 캐시 가능 동적
시스템 프롬프트 ~20-25K 예 아니오
도구 정의 14-17K 예 아니오
MEMORY.md ~500-1K 아니오 아니오
대화 ~900-950K 아니오 예
응답 예비 ~8-10K 아니오 예

Pre-API Call (4단계, 매 요청마다 순차 실행)

1. Snip Compact - 좀비 메시지 제거

대화 스트림에서 더 이상 참조되지 않는 "좀비" 메시지를 식별하여 제거하는, 일종의 메시지 가비지 컬렉션입니다. 비용이 거의 없으면서도 불필요한 토큰 점유를 방지하는 가장 경량의 첫 단계입니다. 대화가 길어질수록 자연스럽게 쌓이는 orphaned 메시지를 정리하여, 이후 단계의 부담을 줄입니다.

2. Microcompact - 도구 결과 삭제

두 가지 경로로 작동합니다. 시간 기반: 마지막 어시스턴트 메시지 이후 임계값(기본 60분)을 초과하면 서버 측 프롬프트 캐시가 만료되었을 가능성이 높으므로, 어차피 전체 프리픽스가 재작성됩니다. 이때 오래된 도구 결과를 함께 제거합니다. 캐시 기반: 프롬프트 캐시가 활성 상태일 때는 cache_edits API를 사용하여 캐시 프리픽스를 보존하면서 도구 결과 블록만 정밀 삭제합니다. 핵심 설계 원칙: 도구 결과는 디스크에서 언제든 다시 읽을 수 있으므로, 삭제해도 정보가 영구적으로 소실되지 않습니다.

3. Context Collapse - 단계적 대화 축소

경량 Microcompact과 비용이 큰 Auto Compact 사이의 중간 지대입니다. 오래된 대화 세그먼트를 점진적으로 축소하되, 두 단계의 임계값으로 작동합니다: 컨텍스트 90% 도달 시 요청을 차단하지 않고 기회적으로 collapse를 commit하고, 95%에서는 collapse가 완료될 때까지 차단합니다. 전체 요약보다 더 많은 대화 구조를 보존하면서도 의미 있는 토큰 절감을 달성합니다.

4. Auto Compact - 세션 메모리 + 모델 기반 요약

가장 무거운 사전 압축 단계로, 두 가지 하위 전략이 있습니다. 먼저 세션 메모리 추출을 시도하여 주요 결정, 발견된 버그, 사용자 선호도를 MEMORY.md 포인터(~150자)로 추출합니다. 이것만으로 충분하면 비용이 큰 API 호출을 회피할 수 있습니다. 부족하면 전체 대화를 Claude에 전송하여 의미론적 요약을 수행하며 70-85%의 극적인 토큰 절감을 달성합니다. 연속 3회 실패 시 circuit breaker가 작동하여 API 낭비를 방지합니다. Post-compact 복구로 최근 파일(최대 5개), 플랜 파일, 호출된 스킬을 재주입하여 작업 컨텍스트를 복원합니다.

Post-API Call (2단계, 413/미디어 에러 시 반응)

5. Collapse Drain - Collapse 큐 일괄 적용

API가 413 Prompt Too Long을 반환하면, 먼저 비용이 낮은 방법을 시도합니다. Stage 3에서 준비해둔 Context Collapse 큐를 모두 일괄 적용하여 즉시 토큰을 확보합니다. Reactive Compact보다 훨씬 저비용이며 프롬프트 캐시를 보존할 수 있어 먼저 시도됩니다. 이것만으로 충분히 토큰이 확보되면, 더 파괴적인 다음 단계 없이 요청을 재시도합니다.

6. Reactive Compact - 긴급 압축

Collapse Drain이 실패하거나 미디어 크기 초과 에러 시 작동하는 최후의 수단입니다. 시스템 프롬프트 + MEMORY.md + 마지막 2턴만 유지하고 나머지를 모두 폐기하며, 이미지/PDF도 제거합니다. 정보 손실이 치명적이지만 MEMORY.md 포인터는 완벽히 보존되므로 모델이 중요한 정보에 여전히 접근할 수 있습니다. 스트리밍 모드로 재시도하여 메모리 버퍼링을 회피하며, 약 1/10,000+ 세션에서만 나타나는 비상 탈출구입니다. 완전히 포기하는 것보다 낫다는 설계 철학을 반영합니다.

☍ Budget

Token Budget Continuation

압축에도 불구하고 모델이 복잡한 작업에서 응답 중간에 max_tokens 제한에 도달하는 경우를 처리하는 synthetic message injection과 3단계 복구 경로입니다. 75-99% 성공률로 불완전한 응답을 자동으로 복구합니다.

자동 계속

응답이 중단되면 시스템이 자동으로 "계속해주세요"라는 숨겨진 메시지를 보냅니다. 사용자는 연속된 응답으로 봅니다.

복구 방법

단계 조건 성공률 정보
1단계 제한 도달 75% 모두 보존
2단계 여전히 미완료 90% MEMORY.md + 최근 2 턴
3단계 여전히 미완료 99% 실시간 스트리밍
핵심: 대부분 1단계에서 끝납니다. 단순히 제한을 높이면 해결되는 경우가 많습니다. 3단계는 거의 필요 없습니다.

응답 종료 감지

API에서 응답이 제대로 끝났는지 확인하는 방법이 중요합니다. 올바른 위치를 확인하지 않으면 무한 대기에 빠질 수 있습니다.

1. Synthetic Message Injection

설정 가능한 토큰 임계값에 도달하면 모델의 응답이 불완전한 상태입니다. 시스템이 "계속 남은 변경사항을 진행하세요"라는 synthetic user message를 투명하게 주입하여 모델이 작업을 이어서 완료하도록 유도합니다. 사용자는 단일 연속적인 응답으로 경험하며, 실제로는 여러 API 호출이 자동으로 연결됩니다.

2. Stage 1: Max Tokens Escalation

모델이 max_tokens로 중단되면 max_tokens를 약 8K에서 약 64K로 에스컬레이션하여 재시도합니다. 불완전한 응답의 약 75%가 이 단계에서 해결되며, 전체 컨텍스트가 보존되므로 정보 손실이 0입니다. 이 간단한 방법이 대부분 충분하다는 점이 전체 복구 전략의 핵심 인사이트입니다.

3. Stage 2: Reactive Compaction

Stage 1 에스컬레이션이 응답을 완료하지 못했을 때 발동합니다. Layer 4 (Reactive Compact)를 적용하여 시스템 프롬프트 + MEMORY.md + 마지막 2 턴만 유지 후 재시도합니다. 나머지 경우의 약 90%가 여기서 해결되며, 대화 이력은 상당히 손실되지만 MEMORY.md 포인터는 완벽히 보존되므로 모델이 여전히 중요한 정보에 접근할 수 있습니다.

4. Stage 3: Streaming Fallback

Stage 2가 여전히 불완전할 때 마지막 수단입니다. 스트리밍 응답 모드를 활성화하여 전체 응답을 메모리에 버퍼링하지 않고 token-by-token으로 점진적으로 처리합니다. 메모리 압력을 방지하고, 사용자는 터미널에서 line-by-line으로 실시간 피드백을 받습니다. 성공률 약 99%이지만, Stage 1과 2가 대부분의 경우를 처리하므로 이 단계는 극히 드뭅니다(약 1/100,000 세션).

5. Session Quality Impact

Budgeting 없는 시나리오에서는 짧은 세션(10턴 미만)은 좋지만 중간(10-50턴)부터 성능 저하, 100+ 턴은 사용 불가능합니다. Budgeting 시스템이 있으면 100+ 턴도 기능적으로 유지됩니다. 6-Stage 컨텍스트 압축(Snip, Microcompact, Context Collapse, Auto Compact + 반응형 Collapse Drain, Reactive Compact) + Synthetic 메시지 + 3-Stage 복구 + Self-healing 메모리의 조합으로 초장시간 세션도 안정적입니다.

⚠ Recovery

Stop Reason & Recovery Handling

Streaming API의 미묘한 stop_reason 처리와 max_tokens에 도달했을 때의 복구 전략입니다. 이 구현 세부사항을 놓치면 무한 대기 또는 응답 손실이 발생하므로, Claude Code의 오랜 세션 안정성을 지탱하는 핵심입니다.

올바른 위치 확인

❌ 잘못된 방법:
for await (const event of response) {
  if (event.type === 'content_block_stop') {
    console.log(event.stop_reason); // ← undefined!
  }
}

✅ 올바른 방법:
for await (const event of response) {
  if (event.type === 'message_delta') {
    console.log(event.delta.stop_reason); // ← "end_turn" or "max_tokens"
  }
}

1. Stop Reason in message_delta

핵심 구현 세부사항: Anthropic API 스트리밍에서 실제 stop_reason은 content_block_stop 이벤트가 아닌 message_delta 이벤트에만 포함됩니다. 공개 문서에 잘 기록되어 있지 않지만, 실제 테스트를 통해 드러난 사실입니다. 이를 놓치면 루프가 제대로 종료되지 않아 무한 대기 또는 timeout 기회 손실이 발생합니다.

2. Stop Reason Types

stop_reason 필드는 정확히 세 값을 가지며, 각각 다른 처리 경로가 필요합니다:
- end_turn: 응답이 자연스럽게 완료 (정상 흐름)
- tool_use: 도구 호출이 발생, 실행 후 재호출 필요
- max_tokens: 응답이 강제 절단됨, 복구 전략 트리거

3. Max Tokens Detection

stop_reason === 'max_tokens'일 때 응답이 불완전하게 절단되었음을 나타냅니다. 이를 감지하면 즉시 Stage 1 복구(max_tokens 8K → 64K 에스컬레이션)를 자동 트리거합니다. 응답이 자연스럽게 끝났는지(end_turn) 또는 강제 중단되었는지(max_tokens) 구분하는 것이 전체 복구 전략의 기초이며, 이 단순한 판단이 대부분의 불완전한 응답을 해결합니다.

4. Tool Use Continuation

tool_use일 때 도구 호출이 대기 중이지만 아직 실행되지 않았음을 의미합니다. 정상 흐름으로 다음 순서를 따릅니다:
(1) 도구 실행 (병렬 또는 순차)
(2) 결과를 대화에 첨부
(3) 모델에 재호출하여 계속 진행
Recovery가 필요하지 않으며, 모델이 자연스럽게 다음 단계로 진행합니다.

5. Infinite Wait Prevention

content_block_stop에서 stop_reason을 확인하면 undefined이므로 루프가 제대로 종료되지 않아 무한 대기 상태에 빠집니다. 또한 timeout 기회를 놓쳐 응답이 완전히 손실될 수 있습니다. 반드시 message_delta 이벤트에서만 확인해야 합니다. 이는 안정적인 streaming 파싱과 recovery cascade의 기초이며, Claude Code의 오랜 세션 안정성을 지탱하는 핵심 구현 세부사항입니다.
