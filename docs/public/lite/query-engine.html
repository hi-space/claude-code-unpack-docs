Query Engine

QueryEngine Deep Dive

Claude Code의 핵심 실행 루프. 사용자 입력을 받아 API 호출, 도구 실행, 컨텍스트 관리를 반복하며 최종 응답을 생성합니다.

Streaming-First Design
5-Layer Compaction
3 Recovery Paths
Token Budget Continuation
⚡ Features

Key Features

QueryEngine의 핵심 설계 원칙과 최적화 기법들

1. Streaming-First Design

StreamProcessor는 Server-Sent Events (SSE)로 수신한 API 응답을 청크 단위로 처리하며, 도구 호출 감지 즉시 실행 시작으로 head-of-line blocking 제거. 모델이 여전히 응답을 생성 중일 때 도구들은 병렬로 실행되어 지각된 레이턴시를 극적으로 감소시킵니다.

2. Stop Reason Handling

StreamProcessor는 Anthropic API의 SSE 스트림을 처리할 때 미묘한 차이를 정확히 구분합니다: content_block_stop 이벤트는 개별 콘텐츠 블록 완료, 실제 stop_reason(end_turn/tool_use)은 message_delta 이벤트에서만 나타납니다. 이 구분을 놓치면 도구 호출 종료 시점 판단 오류 발생.

3. Recovery Paths (max_tokens)

max_tokens에 도달하면 3가지 계층화된 복구 경로를 순차적으로 시도합니다:

  • (1) max_output_tokens를 8K → 16K → 64K로 에스컬레이션
  • (2) Reactive Compact 모드로 컨텍스트 극도로 압축 후 재시도
  • (3) 스트리밍 체계 재설정

☃ Context Compaction & Token Budget

QueryEngine은 토큰 예산 초과 시 5-Layer Context Compaction과 Token Budget Continuation을 활용합니다. 자세한 내용은 Memory → Context Compaction을 참조하세요.

⚡ Core

Core Execution Loop

QueryEngine은 메모리 로드에서 시작하여 반복적으로 메시지를 전송하고, 응답을 파싱하며, 도구를 실행하고, 컨텍스트를 관리하는 주요 루프입니다.

Conversation Loop
flowchart TB
    Start([세션 시작]) --> LoadMem["메모리 로드<br/>MEMORY.md 파싱, 세션 컨텍스트 로드"]
    LoadMem --> AssemblePrompt["시스템 프롬프트 조립<br/>캐시 가능 접두사 + 세션 접미사 구성"]
    AssemblePrompt --> CheckCache{"캐시 경계<br/>변경됨?"}
    CheckCache -->|"아니오"| UseCache["캐시된 접두사 사용<br/>(14-17K 토큰 절약)"]
    CheckCache -->|"예"| Rebuild["전체 프롬프트 재구성<br/>(캐시 누락 페널티)"]

    UseCache --> Loop
    Rebuild --> Loop

    subgraph Loop["대화 루프"]
        SendReq["API 메시지 전송<br/>시스템 프롬프트 +<br/>대화 이력 + 도구 스키마"]
        SendReq --> Stream["스트리밍 응답 처리<br/>SSE 청크 수신"]
        Stream --> Parse{"응답 타입?"}

        Parse -->|"text"| RenderText["텍스트 렌더링<br/>터미널 UI 업데이트"]
        Parse -->|"tool_use"| HandleTool["도구 호출 처리"]

        HandleTool --> PermCheck{"권한 검사<br/>(3계층 파이프라인)"}

        PermCheck -->|"레이어 1: 허용 목록"| AutoAllow["자동 허용<br/>(Read, Glob, Grep...)"]
        PermCheck -->|"레이어 2: 사용자 규칙"| UserRules["설정의 항상 허용<br/>규칙 확인"]
        PermCheck -->|"레이어 3: 분류기"| RunClassifier["AI 분류기로<br/>위험도 판단"]

        RunClassifier -->|"안전함"| AutoAllow
        RunClassifier -->|"위험함"| AskUser["사용자에게<br/>승인/거부 요청"]

        AskUser -->|"승인됨"| AutoAllow
        AskUser -->|"거부됨"| DenyResult["대화에 거부 메시지 반환"]

        AutoAllow --> DispatchTool["도구 실행<br/>검증된 파라미터로 호출"]

        DispatchTool --> ToolResult["도구 결과"]
        DenyResult --> AppendResult
        ToolResult --> AppendResult["대화 기록에<br/>추가"]

        AppendResult --> CheckBudget{"토큰 예산<br/>초과?"}
        CheckBudget -->|"예"| Compress["오래된 메시지 압축"]
        CheckBudget -->|"아니오"| NextIter

        Compress --> NextIter["루프 계속"]
        NextIter --> SendReq

        RenderText --> CheckStop{"종료 조건?"}
        CheckStop -->|"응답 완료"| EndLoop([루프 종료])
        CheckStop -->|"도구 호출"| HandleTool
    end

    EndLoop --> UpdateMem["새로운 사실이 발견된 경우<br/>MEMORY.md 업데이트"]

캐시 경계 관리

시스템 프롬프트는 캐시 가능 접두사(ID, 도구 정의, 안전 규칙 등 고정 블록)와 세션 접미사(git 상태, MCP 스키마 등 가변 블록)로 분리됩니다. 접두사가 변경되지 않으면 캐시를 재사용하여 매 API 호출마다 14-17K 토큰을 절약하며, 캐시 히트 시 약 90%의 비용 할인이 적용됩니다.

3-Layer 권한 파이프라인

모든 도구 호출은 실행 전 3단계 권한 검사를 통과해야 합니다. Layer 1: 읽기 전용 도구(Read, Glob, Grep)는 고정 허용 목록으로 자동 승인. Layer 2: 사용자가 설정한 허용/거부 규칙 확인. Layer 3: 이전 단계에서 결정되지 않은 호출은 AI 분류기(Sonnet)가 위험도를 판단하여 자동 승인 또는 사용자에게 확인을 요청합니다.

토큰 예산 자동 관리

대화가 길어지면 토큰 예산 초과를 감지하여 오래된 메시지부터 자동으로 압축합니다. 압축된 메시지는 핵심 내용만 유지하는 요약으로 대체되어, 100턴 이상의 긴 세션에서도 컨텍스트 품질을 유지하면서 대화를 지속할 수 있습니다.

⚡ Flow

Query Loop Flow

사용자 입력부터 최종 응답까지의 간소화된 흐름으로 컨텍스트 압축, API 호출, 도구 실행, 복구 경로를 보여줍니다.

Query Execution Pipeline
flowchart TB
    INPUT["사용자 입력"] --> COMPACT["컨텍스트<br/>압축"]
    COMPACT --> CALL["모델 호출<br/>(스트리밍)"]
    CALL --> STREAM["스트리밍 도구 처리<br/>(도구 호출 수신)"]
    STREAM --> EXEC["도구 실행<br/>(병렬/순차)"]
    EXEC --> CHECK{"종료 조건?"}
    CHECK -->|"도구 호출"| ATTACH["결과를 대화에<br/>첨부"]

    CHECK -->|"응답 완료"| DONE["응답<br/>완료"]
    CHECK -->|"토큰 초과"| RECOVER["복구:<br/>8K→64K 확장<br/>또는 긴급 압축"]
    RECOVER --> COMPACT

스트리밍 도구 실행

StreamProcessor는 SSE 스트림에서 도구 호출 블록을 감지하는 즉시 실행을 시작합니다. 모델이 여전히 다음 응답을 생성하는 동안 도구가 병렬로 실행되므로, 순차 처리 대비 지각 레이턴시가 크게 감소합니다. content_block_stop(개별 블록 완료)과 message_delta의 stop_reason(전체 응답 완료)을 정확히 구분하여 종료 시점을 판단합니다.

복구 캐스케이드

max_tokens에 도달하면 3단계 자동 복구가 순차적으로 시도됩니다. (1) max_output_tokens를 8K → 16K → 64K로 에스컬레이션하여 출력 공간 확보. (2) Reactive Compact 모드로 컨텍스트를 극도로 압축한 후 재시도. (3) 스트리밍 체계 자체를 리셋합니다. 대부분의 경우 1단계에서 해결되며, 3단계까지 도달하는 경우는 드뭅니다.
