도구 시스템 개요

Claude Code의 도구 시스템은 23+ 빌트인 도구와 MCP 통합을 통해 파일 접근, 코드 실행, 웹 검색 등 다양한 기능을 제공합니다.

핵심 인사이트

1. Fail-Closed Defaults

모든 새로운 도구는 기본적으로 거부/순차 모드로 생성됩니다. 개발자가 명시적으로 권한과 동시성을 활성화해야 하므로, 설정을 빼먹어도 자동으로 가장 안전한 상태가 됩니다. 이 'fail-closed' 원칙이 보안 사고를 원천 차단하는 핵심 설계 철학입니다.

2. 3-Layer Permission Check

도구 실행 전 3단계 권한 검증이 수행됩니다: 입력 스키마 검증, 도구별 커스텀 권한 확인, 전역 권한 정책 적용. 각 레이어가 독립적으로 거부할 수 있어 심층 방어(defense-in-depth)를 구현하고, 한 단계의 설정 실수가 전체 시스템을 위협하지 않도록 합니다.

3. Deferred Tool Loading

자주 사용하지 않는 도구는 초기 프롬프트에 전체 스키마를 포함하지 않고, 이름만 노출합니다. 모델이 해당 도구를 사용하려 할 때 동적으로 스키마를 로드하므로, 초기 프롬프트 크기를 약 3-5K 토큰 절감하여 응답 속도와 비용 효율을 개선합니다.

4. Result Persistence

도구 실행 결과가 큰 경우 전체 결과를 컨텍스트에 넣지 않고 디스크에 저장합니다. 모델에게는 요약(preview)만 전달하여 컨텍스트 윈도우를 효율적으로 사용합니다. 큰 파일 읽기, 방대한 검색 결과, 긴 셸 출력 등에서 특히 중요한 전략입니다.

5. Prompt Cache Stability

도구 목록을 API에 전달할 때 항상 정렬된 순서를 유지합니다. Anthropic API의 prompt caching은 도구 목록의 순서가 바뀌면 캐시 키가 달라져 히트율이 떨어지므로, 동일한 도구 세트는 항상 동일한 캐시 키를 생성하도록 보장합니다. 전체 도구 스키마는 약 14-17K 토큰을 차지합니다.

요구사항: 파일 타입에 맞는 LSP 서버 구성 필요

텍스트 검색과의 차이

Grep/Glob이 단순 텍스트 패턴 매칭이라면, LSP는 Language Server와 연결하여 심볼의 의미적 관계를 파악합니다. 변수 이름이 여러 스코프에서 중복되어도 정확한 정의 위치를 추적하며, 타입 정보와 함수 시그니처를 기반으로 코드 구조를 이해합니다. 대규모 코드베이스에서 리팩토링 영향 범위를 정확히 판단할 때 특히 유용합니다.

주요 활용 시나리오

Go-to-Definition: 함수/변수가 어디서 정의되었는지 즉시 추적. Find References: 특정 심볼이 코드베이스 전체에서 어디서 사용되는지 파악하여 수정 시 영향 범위 확인. Hover: 심볼의 타입 정보와 문서를 조회. Diagnostics: 컴파일 에러, 타입 불일치 등을 실시간으로 감지하여 코드 품질을 사전에 확보합니다.
